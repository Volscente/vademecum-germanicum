## Architecture

```txt
.env
  │  GEMINI_API_KEY=...
  │  LLM_MODEL=gemini/gemini-2.0-flash
  │
  ▼
docker-compose.yml
  │  environment:
  │    GEMINI_API_KEY: ${GEMINI_API_KEY}
  │    LLM_MODEL: ${LLM_MODEL}
  │
  ▼
backend container
  │  os.getenv("GEMINI_API_KEY")
  │  os.getenv("LLM_MODEL")
  │
  ▼
LiteLLM (reads GEMINI_API_KEY automatically)
  │
  ▼
PydanticAI agent (uses LiteLLM as model provider)
```

### Why LLM_MODEL as an environment variable

The RFC specifies model-provider abstraction via LiteLLM. Externalising the model identifier (`gemini/gemini-2.0-flash`) as `LLM_MODEL` allows swapping providers without code changes — only `.env` needs updating. LiteLLM resolves the model string and picks the correct API key env var automatically (e.g. `GEMINI_API_KEY` for Gemini models).

### Environment Variables

| Variable        | Required | Default                    | Description                                                            |
| --------------- | -------- | -------------------------- | ---------------------------------------------------------------------- |
| `GEMINI_API_KEY` | Yes      | —                          | Google AI Studio API key for Gemini Flash access                       |
| `LLM_MODEL`     | No       | `gemini/gemini-2.0-flash`  | LiteLLM model identifier; change to switch provider without code edits |

`GEMINI_API_KEY` is read automatically by LiteLLM when a `gemini/*` model is selected. No custom loading code is needed.

### docker-compose.yml Changes

Add to the `backend` service `environment` block:

```yaml
environment:
  DATABASE_URL: postgresql://${POSTGRES_USER}:${POSTGRES_PASSWORD}@${POSTGRES_HOST_DOCKER}:${POSTGRES_PORT}/${POSTGRES_DB}
  GEMINI_API_KEY: ${GEMINI_API_KEY}
  LLM_MODEL: ${LLM_MODEL:-gemini/gemini-2.0-flash}
```

The `LLM_MODEL` variable uses a shell default so it can be omitted from `.env` during development.
